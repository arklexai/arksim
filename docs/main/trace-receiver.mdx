## Overview

Some agents handle tool calling internally (e.g. OpenAI Agents SDK, LangChain) and only return text. arksim can capture these tool calls automatically so the evaluator can score them, without requiring changes to the agent's return value.

There are two ways to provide tool calls to arksim:

1. **Explicit**: Return `AgentResponse(content=..., tool_calls=[...])` from `execute()`. No extra config needed.
2. **Automatic capture**: Add `trace_receiver: { enabled: true }` to your config. arksim captures tool calls via the SDK's tracing interface. The agent just returns text.

If both are active, arksim deduplicates by tool call ID first, then by `(name, arguments)` signature.

---

## Automatic Capture (Python Connector)

For agents using the OpenAI Agents SDK with the Python connector, add two things:

**1. Register the processor at module level in your agent file:**

```python
from agents.tracing import add_trace_processor
from arksim.tracing.openai import ArksimTracingProcessor

add_trace_processor(ArksimTracingProcessor())
```

**2. Enable the trace receiver in your config:**

```yaml
trace_receiver:
  enabled: true
  wait_timeout: 5
```

That's it. Your agent's `execute()` method just returns a string:

```python
class MyAgent(BaseAgent):
    async def execute(self, user_query, **kwargs):
        result = await Runner.run(
            self._agent,
            input=[{"role": "user", "content": user_query}],
        )
        return result.final_output
```

The simulator sets routing context (conversation ID, turn ID) before each call to `execute()`. When the SDK fires `on_span_end`, the processor reads this context and injects tool calls into the receiver's buffer. No wrapping, no manual context management.

The module loader caches modules by file path, so `add_trace_processor` runs exactly once regardless of how many conversations the simulator creates.

See `examples/customer-service/traced_agent.py` for a complete working example.

Requires `pip install openai-agents`.

## Connector Support

| Connector | Tool Call Capture | How to use |
|-----------|------------------|------------|
| Python connector | Supported | `ArksimTracingProcessor` (see above) |
| A2A | Planned | Protocol has `task_id` / `context_id` for correlation |
| Chat Completions | Planned | W3C Trace Context for correlation |

<Info>
A2A and Chat Completions connectors do not yet support automatic tool call capture. If you need tool call evaluation for these connectors, use the Python connector with `ArksimTracingProcessor`.
</Info>

## Configuration Reference

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | bool | `false` | Enable automatic tool call capture |
| `host` | string | `127.0.0.1` | Bind address (used by upcoming cross-connector support) |
| `port` | int | `4318` | Port number (used by upcoming cross-connector support) |
| `wait_timeout` | float | `5.0` | Max seconds to wait for tool calls after each agent turn |

For the Python connector, only `enabled` and `wait_timeout` are relevant. The `host` and `port` fields are used by the OTLP/HTTP receiver for upcoming A2A and Chat Completions support.

## Metadata Keys

arksim passes a `metadata` dict to your agent's `execute()` via `kwargs["metadata"]`:

| Key | Type | Description |
|-----|------|-------------|
| `chat_id` | `str` | Conversation ID assigned by `agent.get_chat_id()` |
| `turn_id` | `int` | Zero-based turn index within the conversation |

These keys are always present regardless of whether the trace receiver is enabled. Trace routing context is passed to the processor automatically by the simulator, not through metadata.

---

## Supported Tool Call Attributes

The receiver recognizes tool calls from two OTel attribute conventions:

**OTel GenAI semconv** (recommended):
- `gen_ai.tool.name`, `gen_ai.tool.call.arguments`, `gen_ai.tool.call.result`, `gen_ai.tool.call.id`

**OpenInference (Arize)**:
- `tool.name`, `tool_call.function.arguments` or `tool.parameters`, `output.value`, `tool_call.id` or `tool.id`

Only spans with a recognized tool name attribute or a span name starting with `execute_tool` (followed by a space and the tool name) are converted. Other spans (HTTP, DB, parent) are filtered out.

---

## Deduplication

When both `AgentResponse` and the trace receiver provide tool calls for the same turn, arksim merges and deduplicates:

1. **By ID**: tool calls with matching IDs are deduplicated (AgentResponse wins)
2. **By signature**: tool calls with matching `(name, arguments)` are deduplicated (AgentResponse wins)
3. **Unique**: tool calls from either source that don't match anything are included

This allows both capture paths to coexist safely.
