## Overview

The trace receiver captures tool calls from agents that handle tools internally. Instead of requiring agents to return tool calls in `AgentResponse`, arksim captures them automatically via the SDK's tracing interface and attaches them to `Message.tool_calls` for evaluation. For same-process agents, this is zero-setup. For cross-process agents, an OTLP/HTTP endpoint accepts OTel span exports.

This is useful when:
- Your agent uses an SDK that handles tool calling internally (e.g. OpenAI Agents SDK, LangChain)
- You cannot modify the agent's return value to include `AgentResponse`
- You want to capture tool calls from an agent running in a separate process

---

## Configuration

Add a `trace_receiver` block to your simulation config:

```yaml
trace_receiver:
  enabled: true
  host: 127.0.0.1   # bind address
  port: 4318         # IANA-assigned OTLP/HTTP standard port
  wait_timeout: 5    # seconds to wait for traces after each agent response
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | bool | `false` | Enable the trace receiver |
| `host` | string | `127.0.0.1` | Bind address (`0.0.0.0` for containers, trusted networks only) |
| `port` | int | `4318` | Port number (1-65535) |
| `wait_timeout` | float | `5.0` | Max seconds to wait for traces per turn |

## Capture Paths

arksim supports three ways to capture tool calls. They can be combined freely, and arksim deduplicates by tool call ID first, then by `(name, arguments)` signature.

### 1. AgentResponse (no trace receiver needed)

The agent returns `AgentResponse(content=..., tool_calls=[...])` from `execute()`. This is the simplest path and requires no extra infrastructure.

### 2. Direct injection (same process)

When the agent runs in the same process as arksim, the simulator passes routing context to the `TracingProcessor` via `contextvars`. The processor's `on_span_end` callback reads the context and calls `receiver.submit_tool_calls()` to inject tool calls directly into the receiver's buffer, bypassing HTTP entirely.

This is the recommended path for same-process agents because it avoids event loop contention between the OTel HTTP exporter and the async receiver. No wrapping or manual context management is needed in the agent.

### 3. HTTP export (separate process)

When the agent runs in a separate process, it exports OTel spans via `OTLPSpanExporter` to `POST /v1/traces` on the receiver's port. The receiver accepts both protobuf (`application/x-protobuf`) and JSON payloads.

```bash
pip install arksim[otel]  # for protobuf support
```

Without `arksim[otel]`, protobuf payloads are rejected with HTTP 415. JSON payloads work without extra dependencies.

For HTTP agents, text-only turns (no tool calls) block for the full `wait_timeout` since the receiver cannot distinguish "no spans yet" from "turn complete with no tool calls." The `ArksimTracingProcessor` handles this automatically for same-process agents via `signal_turn_complete()`. For HTTP agents, consider using a lower `wait_timeout` (e.g., 2s) if most turns produce text-only responses.

## Connector Support

Trace correlation depends on the agent connector type:

| Connector | Trace Correlation | Mechanism |
|-----------|------------------|-----------|
| Python connector | Supported | `ArksimTracingProcessor` + `contextvars` |
| A2A | Not yet supported | Protocol has `task_id` / `context_id` (planned) |
| Chat Completions | Not yet supported | - |

<Info>
A2A and Chat Completions agents running externally do not yet have automatic trace correlation. If you need tool call capture, use the Python connector with `ArksimTracingProcessor` instead.
</Info>

## Agent Instrumentation

### Using ArksimTracingProcessor (recommended)

For agents built with the OpenAI Agents SDK, arksim ships a ready-to-use `TracingProcessor` that captures tool calls automatically.

**With `arksim simulate` (zero setup):** The simulator registers the processor and sets routing context via `contextvars`. Your agent needs no tracing code at all:

```python
class MyAgent(BaseAgent):
    async def execute(self, user_query, **kwargs):
        result = await Runner.run(self._agent, input=input_list)
        return result.final_output
```

Just enable the trace receiver in your config and run normally. See `examples/customer-service/traced_agent.py` for a complete example.

**Standalone (outside `arksim simulate`):** Register the processor with the SDK directly and use `.trace()` for routing context:

```python
from agents.tracing import add_trace_processor
from arksim.tracing.openai import ArksimTracingProcessor
from arksim.tracing.receiver import TraceReceiver

processor = ArksimTracingProcessor()
add_trace_processor(processor)

async with TraceReceiver(port=4318) as receiver:
    async with processor.trace(conversation_id="conv-1", turn_id=0, receiver=receiver):
        result = await Runner.run(agent, input=input_list)
    tool_calls = await receiver.wait_for_traces("conv-1", 0)
```

Requires `pip install openai-agents`.

### Metadata keys

arksim passes a `metadata` dict to your agent's `execute()` via `kwargs["metadata"]`:

| Key | Type | Description |
|-----|------|-------------|
| `chat_id` | `str` | Conversation ID assigned by `agent.get_chat_id()` |
| `turn_id` | `int` | Zero-based turn index within the conversation |

These keys are always present regardless of whether the trace receiver is enabled. Trace routing context (conversation ID, turn ID, receiver reference) is passed to the processor automatically via `contextvars`, not through metadata.

### Using raw OTel spans (any framework)

For agents not using the OpenAI Agents SDK, create OTel spans with routing attributes. The `conversation_id` and `turn_id` values come from arksim's metadata dict (passed to your agent's `execute()` via `kwargs["metadata"]`):

```python
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
from opentelemetry.sdk.resources import Resource
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import SimpleSpanProcessor

conversation_id = "your-conversation-id"  # from metadata["chat_id"]
turn_id = 0                               # from metadata["turn_id"]

provider = TracerProvider(
    resource=Resource.create({
        "service.name": "my-agent",
        "arksim.conversation_id": conversation_id,
    })
)
provider.add_span_processor(
    SimpleSpanProcessor(OTLPSpanExporter(endpoint="http://127.0.0.1:4318/v1/traces"))
)
tracer = provider.get_tracer("my-agent")

with tracer.start_as_current_span("execute_tool search_flights") as span:
    span.set_attribute("arksim.turn_id", turn_id)
    span.set_attribute("gen_ai.tool.name", "search_flights")
    span.set_attribute("gen_ai.tool.call.arguments", '{"origin": "NYC"}')
    span.set_attribute("gen_ai.tool.call.result", '{"flights": [...]}')

provider.force_flush()
```

### Routing attributes

| Attribute | Where | Description |
|-----------|-------|-------------|
| `arksim.conversation_id` | Resource or span | Conversation ID from `metadata["chat_id"]` |
| `arksim.turn_id` | Span | Turn index from `metadata["turn_id"]` |

### Supported tool call attributes

The receiver parses tool call data from two OTLP attribute families:

**OTel GenAI semconv** (recommended):
- `gen_ai.tool.name`, `gen_ai.tool.call.arguments`, `gen_ai.tool.call.result`, `gen_ai.tool.call.id`

**OpenInference (Arize)**:
- `tool.name`, `tool_call.function.arguments` or `tool.parameters`, `output.value`, `tool_call.id` or `tool.id`

Only spans with a recognized tool name attribute or the `execute_tool ` span name prefix are converted. Other spans (HTTP, DB, parent) are filtered out.

## Same-Process Considerations

When the traced agent and arksim run in the same process:

<Warning>
Use `num_workers: 1` in your config. The OpenAI Agents SDK's global trace context does not reliably propagate across many concurrent async traces. This limitation applies only to same-process tracing. Agents in separate processes can use any parallelism level.
</Warning>

The direct injection path (`submit_tool_calls`) avoids the HTTP deadlock that would otherwise occur when the OTel exporter and the trace receiver share an event loop. The simulator automatically passes routing context to the processor via `contextvars` when the trace receiver is enabled.

```yaml
# config_traced.yaml
num_workers: 1  # required for same-process tracing

trace_receiver:
  enabled: true
  port: 4318
  wait_timeout: 5
```

---

## Deduplication

When both `AgentResponse` and the trace receiver provide tool calls for the same turn, arksim merges and deduplicates:

1. **By ID**: tool calls with matching IDs are deduplicated (AgentResponse wins)
2. **By signature**: tool calls with matching `(name, arguments)` are deduplicated (AgentResponse wins)
3. **Unique**: tool calls from either source that don't match anything are included

This allows both capture paths to coexist safely.
