## Overview

Configure your agent's API connection. The `agent_config.json` file tells arksim how to connect to your agent API. Scenarios (goals, user profiles, and any scenario-specific knowledge) are defined separately in `scenarios.json` and referenced from your run config (e.g. `config.yaml`).

### What you need

In your example or project directory you need at least:

- **`agent_config.json`** â€“ API and agent configuration (endpoint, headers, body, agent type). Described below.

Your run configuration (e.g. `config.yaml`) points to this file via `agent_config_file_path` and to your scenarios via `scenario_file_path`. See the [quickstart](/quickstart) and example docs for full setup.

## Agent Configuration

The `agent_config.json` file configures how arksim connects to your conversational agent API.

### Configuration Fields

| Field        | Type   | Required | Description                                                                            |
| ------------ | ------ | -------- | -------------------------------------------------------------------------------------- |
| `agent_name` | String | Yes      | Unique identifier for your agent (e.g. lowercase, no spaces)                           |
| `agent_type` | String | Yes      | One of `chat_completions` or `a2a`. See [API Connection Types](#api-connection-types). |
| `api_config` | Object | Yes      | Connection details. See [API Connection Types](#api-connection-types).                 |

### API Connection Types

arksim supports two agent types for API connections:

<Tabs>
  <Tab title="Chat Completions">
    **Type:** `chat_completions`

    Connects to any OpenAI-compatible chat completions endpoint.

    #### Required fields

    - `endpoint`: URL of the chat completions API
    - `headers`: HTTP headers (e.g. `Content-Type`, `Authorization`)
    - `body`: Request body; must include a `messages` array

    #### Placeholders and variables

    - `{chat_id}`: Can be included in `body.messages[*].content`; replaced at runtime
    - `${ENV_VAR}`: Supported in header values for secrets

    #### Example

    ```json
    {
      "agent_type": "chat_completions",
      "agent_name": "chat_completions_example",
      "api_config": {
        "endpoint": "http://localhost:7050/chat/completions",
        "headers": {
          "Content-Type": "application/json",
          "Authorization": "Bearer ${OPENAI_API_KEY}"
        },
        "body": {
          "model": "gpt-5.1",
          "messages": [
            {
              "role": "system",
              "content": "You are being invoked by arksim user simulator."
            }
          ]
        }
      }
    }
    ```

    #### Example with Chat ID

    ```json
    {
      "agent_type": "chat_completions",
      "agent_name": "chat_completions_with_chat_id",
      "api_config": {
        "endpoint": "http://localhost:7050/chat/completions",
        "headers": {
          "Content-Type": "application/json",
          "Authorization": "Bearer ${OPENAI_API_KEY}"
        },
        "body": {
          "model": "gpt-5.1",
          "messages": [
            {
              "role": "system",
              "content": "You are being invoked with chat_id:{chat_id}"
            }
          ]
        }
      }
    }
    ```

    <Note>
      The `{chat_id}` placeholder is replaced with the actual chat ID during execution.
    </Note>
  </Tab>
  <Tab title="A2A Protocol">
    **Type:** `a2a`

    For agents that speak the Agent-to-Agent (A2A) protocol.

    #### Required fields

    - `endpoint`: A2A agent server URL
    - `headers`: Optional HTTP headers (e.g. for auth)

    #### Example

    ```json
    {
      "agent_type": "a2a",
      "agent_name": "a2a_example",
      "api_config": {
        "endpoint": "http://localhost:9999",
        "headers": {
          "api-key": "${A2A_AUTH_TOKEN}"
        }
      }
    }
    ```
  </Tab>
</Tabs>

### Environment variable support

Both `chat_completions` and `a2a` support environment variable substitution using `${ENV_VAR}` in header values (and, for chat completions, in the endpoint URL where applicable).

**How it works:**

- At runtime, `${ENV_VAR}` is replaced with the value of that environment variable
- If the variable is not set, it is replaced with an empty string
- You can mix static text with variables, e.g. `"Bearer ${API_KEY}"` â†’ `"Bearer sk-abc123..."`

<Tip>
  **Security:** Keep credentials in environment variables and out of committed config files.
</Tip>
