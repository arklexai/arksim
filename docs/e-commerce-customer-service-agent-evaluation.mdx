## Overview

This example walks through running Arksim against a shopping assistant agent built for an e-commerce use case. The agent is designed to help customers with product discovery, orders, returns, and general shopping queries.

The example includes two ready-to-run agent setups you can test against out of the box, and a guide for plugging in your own agent once you're familiar with the setup.

## Scenarios

The example ships with pre-built scenarios in `scenarios.json` representing realistic e-commerce customer interactions. Each scenario defines a simulated user with a distinct persona, goal, and background knowledge drawn from product documentation.

The scenario goals are:

- Return an item as a returning customer without providing order ID or reason (expecting a new baby next week).
- Return an item as a returning customer without providing order ID or reason (moving into a new home or apartment next month).
- Return an item as a returning customer without providing order ID or reason (adopting a pet for the first time next week).
- Return an item as a returning customer without providing order ID or reason (deep-cleaning before guests arrive in two days).
- Return an item as a returning customer without providing order ID or reason (sending a child to college or dorm setup next month).

Scenarios are defined in `scenarios.json` in the example directory and can be edited or extended to reflect your own use case.

## Option 1: OpenAI Agent

This agent calls the OpenAI API directly, with no server setup required.

<Steps>
  <Step title="Set your API key">
    ```bash
    export OPENAI_API_KEY="<YOUR_OPENAI_API_KEY>"
    ```
  </Step>
  <Step title="Run simulation and evaluation">
    Run from the `examples/e-commerce` directory:

    ```bash
    cd examples/e-commerce
    arksim simulate-evaluate config.yaml
    ```
  </Step>
</Steps>

## Option 2: In-house Agent (LangGraph RAG)

This agent is a RAG-based shopping assistant built with LangGraph, backed by product data. It is exposed via a Chat Completions-compatible endpoint on port 8080 at `/chat/completions`.

<Steps>
  <Step title="Select agent config">
    In the example directory, copy `agent_config_chat_completions.json` to `agent_config.json`. If `agent_config.json` already exists, rename or remove it first.

    The chat completions config uses `${AGENT_API_KEY}` for the local server; set it when running the simulator (e.g. `export AGENT_API_KEY="<YOUR_AGENT_API_KEY>"`).
  </Step>
  <Step title="Install agent dependencies">
    <Tabs>
      <Tab title="uv">
        ```bash
        uv venv --python 3.11
        source .venv/bin/activate
        pip install -r examples/e-commerce/chat_completions_server/requirements.txt
        ```
      </Tab>
      <Tab title="conda">
        ```bash
        conda create -n ecommerce_venv python=3.11 -y
        conda activate ecommerce_venv
        pip install -r examples/e-commerce/chat_completions_server/requirements.txt
        ```
      </Tab>
    </Tabs>
    Or from inside the example directory: `pip install -r chat_completions_server/requirements.txt`
  </Step>
  <Step title="Start the agent server">
    Run from the repository root (or the directory that contains the `examples` package). The agent listens on port 8080.

    ```bash
    export OPENAI_API_KEY=<YOUR_OPENAI_API_KEY>
    python -m examples.e-commerce.chat_completions_server.chat_completion_wrapper
    ```
  </Step>
  <Step title="Run simulation and evaluation">
    In a new terminal, run from the `examples/e-commerce` directory:

    ```bash
    cd examples/e-commerce
    export OPENAI_API_KEY="<YOUR_OPENAI_API_KEY>"
    export AGENT_API_KEY="<YOUR_AGENT_API_KEY>"
    arksim simulate-evaluate config.yaml
    ```
  </Step>
</Steps>

## Running with Your Own Agent

To test your own backend agent against these scenarios, follow the comments in `chat_completions_server/chat_completion_wrapper.py` to call your backend instead of the sample agent.

Once your server is running on a Chat Completions-compatible endpoint, point `agent_config.json` at it and run simulation and evaluation from `examples/e-commerce` as above.

<Tip>
  See [Agent Compatibility](/agent_configurations) for a full breakdown of supported protocols and how to configure `agent_config.json`.
</Tip>

## Configuration

The example uses a single config file for both simulation and evaluation. Key settings:

```yaml
# Agent and scenarios (paths relative to config file location)
agent_config_file_path: ./agent_config.json
scenario_file_path: ./scenarios.json

# Simulation
num_conversations_per_scenario: 1
max_turns: 5
output_file_path: ./results/simulation/simulation.json

# Evaluation
output_dir: ./results/evaluation
custom_metrics_file_paths:
  - ./custom_metrics.py
metrics_to_run:
  - faithfulness
  - helpfulness
  - coherence
  - verbosity
  - relevance
  - goal_completion
  - agent_behavior_failure
generate_html_report: true

# Shared
model: gpt-5.1
provider: openai
num_workers: auto
```

The config also defines a `simulated_user_prompt_template` (Jinja) that uses `scenario.agent_context`, `scenario.goal`, `scenario.knowledge`, and `simulation.profile` to drive the simulated user.

<Note>
  The example directory contains `agent_config.json` (default OpenAI), `agent_config_chat_completions.json`, the product knowledge under `chat_completions_server/data/`, and the scenario file `scenarios.json`.
</Note>

## Output

Results are written under the example directory:

| Location                                 | Contents                                                                 |
| ---------------------------------------- | ------------------------------------------------------------------------ |
| `./results/simulation/simulation.json`   | Simulated conversations from the simulation step                         |
| `./results/evaluation/evaluation.json`   | Evaluation results (per-turn and per-conversation scores, unique errors) |
| `./results/evaluation/final_report.html` | Interactive HTML report for browsing and sharing results                 |
