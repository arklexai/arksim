## Overview

This example walks through running Arksim against a customer service agent built for an insurance company. The agent is designed to answer customer questions about insurance products and coverage — including topics like policy details, claims processes, deductibles, and coverage limits.

The example includes two ready-to-run agent implementations you can test against out of the box, and a guide for plugging in your own agent once you're familiar with the setup.

## Scenarios

The example ships with a set of pre-built scenarios representing realistic insurance customer interactions. Each scenario defines a simulated user with a distinct persona, goal, and background knowledge drawn from insurance product documentation.

Sample goals include:

- Asking detailed questions about car insurance coverage options, limits, and deductibles
- Finding out whether a home insurance policy covers water damage from a burst pipe
- Understanding what documents are needed to file an auto insurance claim
- Comparing leasing vs. purchasing options for a service product

Scenarios are located at `./examples/bank-insurance/scenarios/` and can be swapped out or extended to reflect your own use case.

## Example Agent 1: OpenAI API

This agent calls the OpenAI API directly — no server setup required.

<Steps>
  <Step title="Set your API key">
    ```bash
    export OPENAI_API_KEY="<YOUR_OPENAI_API_KEY>"
    ```
  </Step>
  <Step title="Run simulation and evaluation">
    ```bash
    arksim simulate ./examples/bank-insurance/config.yaml
    arksim evaluate ./examples/bank-insurance/config.yaml
    ```
  </Step>
</Steps>

## Example Agent 2: LangGraph RAG Agent

This agent is a RAG-based customer service agent built with LangGraph, backed by an insurance knowledge base. It can be exposed via either the A2A Protocol or a Chat Completions-compatible endpoint.

<Steps>
  <Step title="Install agent dependencies">
    <Tabs>
      <Tab title="uv">
        ```bash
        uv venv --python 3.11
        source .venv/bin/activate
        pip install -r ./examples/bank-insurance/agent_server/requirements.txt
        ```
      </Tab>
      <Tab title="conda">
        ```bash
        conda create -n bank_venv python=3.11 -y
        conda activate bank_venv
        pip install -r ./examples/bank-insurance/agent_server/requirements.txt
        ```
      </Tab>
    </Tabs>
  </Step>
  <Step title="Start the agent server">
    <Tabs>
      <Tab title="A2A">
        Exposes an A2A-compatible agent on port `9999`:

        ```bash
        export OPENAI_API_KEY=<YOUR_OPENAI_API_KEY>
        python -m examples.bank-insurance.agent_server.a2a.server
        ```
      </Tab>
      <Tab title="Chat Completions">
        Exposes an OpenAI Chat Completions-compatible endpoint on port `8888` at `/chat/completions`:

        ```bash
        export OPENAI_API_KEY=<YOUR_OPENAI_API_KEY>
        python -m examples.bank-insurance.agent_server.chat_completions.server
        ```

        When `enable_metadata` is set to `true` in the request body, the following fields are sent alongside each user message: `chat_id`, `user_attributes`, `user_goal`, `knowledge`.
      </Tab>
    </Tabs>
  </Step>
  <Step title="Run simulation and evaluation">
    In a new terminal:

    ```bash
    export OPENAI_API_KEY="<YOUR_OPENAI_API_KEY>"
    arksim simulate ./examples/bank-insurance/config.yaml
    arksim evaluate ./examples/bank-insurance/config.yaml
    ```
  </Step>
</Steps>

## Running with Your Own Agent

To test your own backend agent against these scenarios, you have two options depending on your preferred protocol:

- **Chat Completions**: follow the `#TODO` comments in `agent_server/chat_completions/server.py` to swap in your own backend logic.
- **A2A**: implement your own A2A executor in `agent_server/a2a/`.

Once your server is running, point `agent_config.json` at your endpoint and run simulation and evaluation as normal.

<Tip>
  See Agent Compatibility for a full breakdown of supported protocols and how to configure `agent_config.json`.
</Tip>

## Configuration

```yaml
# Directory containing agent_config.json, knowledge data, and example scenarios
agent_setup_dir: ./examples/bank-insurance

# Where all results are written
output_dir: ./examples/bank-insurance/results

# Number of conversations to simulate
num_conversations: 5

# Maximum turns per conversation
max_turns: 5

# Metrics to run during evaluation
metrics_to_run: [faithfulness, helpfulness, coherence, verbosity, relevance, goal_completion, agent_behavior_failure]

# Paths to .py files defining custom metrics (must implement BaseMetric)
custom_metrics_file_paths: []

# Generate an interactive HTML report
generate_html_report: true

# LLM used to power simulation and evaluation scoring
model: gpt-5.1
provider: openai
num_workers: auto
```

<Note>
  `agent_setup_dir` is the root directory for the example — it contains the `agent_config.json`, the insurance knowledge base, and the pre-built scenario file used to drive the simulated users.
</Note>

## Output

Results are written to `./examples/bank-insurance/results/`:

| File                                     | Contents                                                              |
| ---------------------------------------- | --------------------------------------------------------------------- |
| `agent_performance_per_turn.csv`         | Per-turn metric scores, reasoning, and failure labels                 |
| `agent_performance_per_conversation.csv` | Rolled-up scores and status per conversation                          |
| `unique_errors.csv`                      | Deduplicated failure patterns with descriptions and occurrence traces |
| `final_report.html`                      | Interactive report for browsing and sharing results                   |
