## What is a Scenario?

A scenario is a test case that describes who the simulated user is, what they want to accomplish, and what they already know going into the conversation with the agent.

![Arksim Workflow](/images/scenarios.svg)

## The Three Core Pillars

### Goal

Describes the user's objective, guiding their actions and decisions throughout the conversation.

> "Plan a purchase of a service robot, focusing on delivery timelines and suitability for commercial use."

### User Attributes (Persona)

Defines the user's characteristics, influencing their tone, communication style, and behavior.

> "Alex, 26 years old, living in Chicago, works as an operations analyst at a mid-sized logistics company; analytical, detail-oriented, budget-conscious, and risk-aware."

### Knowledge

Provides context that shapes the simulated user's questions and expectations. Also used during **evaluation** to check whether the agent's responses were accurate and consistent with the source.

> - `adam_robot_spec.md` \
>   The ADAM robot is a 40 kg payload autonomous service robot with 8-hour battery life and \$70,000 price…
> - `purchase_options.pdf` \
>   The payment plans include full purchase, 12-month leasing, and vendor financing…

<Tip>
  Think of a scenario as answering three questions: **Who** is this user? **What** do they want? **What** do they know?
</Tip>

## Why Arksim Scenarios?

Arksim Scenarios let you test and evaluate your agent using realistic, profile-driven simulations of user interactions. Each scenario is built around clearly defined goals and behaviors, enabling structured performance evaluation without requiring a real-world dataset.

By using synthetic users, you can run consistent and reproducible tests across a wide range of interaction types — making it easy to scale evaluations and understand how your agent performs before deploying it to real users.

## Scenario File Structure

A scenario file is a single JSON document containing a list of scenario objects. Each scenario represents one simulated user and one conversation session.

```json
{
  "schema_version": "v1",
  "scenarios": [
    { ... },
    { ... }
  ]
}
```

---

## Schema Reference

<ResponseField name="schema_version" type="string" required>
  Schema version identifier. Use `"v1"`.
</ResponseField>

<ResponseField name="scenarios" type="list[dict]" required>
  List of scenario objects. Each object represents one simulated user and one conversation session.

  <Expandable title="scenario fields">
    <ResponseField name="scenario_id" type="string" required>
      Unique identifier for this scenario.
    </ResponseField>
    <ResponseField name="user_id" type="string" required>
      Unique identifier for the simulated user. Defaults to `scenario_id` if omitted.
    </ResponseField>
    <ResponseField name="goal" type="string" required>
      The user's conversational goal, passed directly into the user simulator prompt. Write it in second person ("You want to...") so it reads naturally as a simulator instruction.
    </ResponseField>
    <ResponseField name="user_attribute" type="dict[string, Any]" required>
      A key-value map representing the traits of a simulated user. Keys and values can be defined freely. These attributes serve as the foundation for generating the user's persona and guiding behavior in evaluations.

      Common attributes used in pre-built scenarios include:

      - **Demographic:** `sex`, `age`, `marital_status`, `household_size`, `location`, `Household_income`, `education`
      - **Personality (Big Five):** `Openness`, `Conscientiousness`, `Extraversion`, `Agreeableness`, `Neuroticism`
      - **Behavioral:** `customer_type`, `discovery_type`, `decision_making_style`, `purchasing_preference`, `spending_behavior`, `loyalty_level`
    </ResponseField>
    <ResponseField name="knowledge" type="list[KnowledgeItem]">
      Knowledge items embedded in the simulated user's context during the conversation. Can be omitted if the simulated user has no specific background information.

      Each `KnowledgeItem` contains:

      - `content` (required) — The text content injected into the simulator prompt as the user's background knowledge.
      - `metadata` — Freeform metadata for labeling and traceability (e.g. `title`, `source`, `doc_id`). Not used by the simulator or evaluator.
    </ResponseField>
    <ResponseField name="origin" type="dict[string, Any]">
      Freeform metadata used to record how a scenario was generated. Not used by the simulator or evaluator — exists purely for provenance tracking.

      Common fields:

      - `target_agent_capability` — The agent capability this scenario was designed to evaluate.
      - `goal_raw` — The original goal before it was adapted to the full contextual goal string.
    </ResponseField>
  </Expandable>
</ResponseField>

## Full Example

```json
{
  "schema_version": "v1",
  "scenarios": [
    {
      "scenario_id": "123456",
      "user_id": "123456",
      "goal": "You want to ask detailed questions about TD Insurance car coverage options in your province, including limits, deductibles, and what's actually covered.",
      "knowledge": [
        {
          "content": "Car Insurance Overview: TD Insurance offers mandatory and optional coverages including third-party liability, collision, and comprehensive...",
          "metadata": {
            "title": "Car Insurance Overview",
            "source": "https://www.tdinsurance.com/products-services/auto-car-insurance",
            "knowledge_type": "car_insurance_overview",
            "doc_id": "car_insurance_overview_001"
          }
        }
      ],
      "user_attribute": {
        "sex": "Female",
        "age": "34",
        "marital_status": "single",
        "household_size": "1",
        "location": "Toronto, ON, Canada",
        "Household_income": "$50,000 - $74,999",
        "education": "Bachelor's degree",
        "Openness": "open to experience",
        "Conscientiousness": "conscientious",
        "Extraversion": "introverted",
        "Agreeableness": "agreeable",
        "Neuroticism": "emotionally stable",
        "customer_type": "prospective customer",
        "discovery_type": "online search",
        "decision_making_style": "analytical",
        "purchasing_preference": "value oriented",
        "spending_behavior": "moderate spending",
        "loyalty_level": "neutral"
      },
      "origin": {
        "target_agent_capability": "Answer questions about TD Insurance products and coverage",
        "goal_raw": "Ask question about TD bank insurance"
      }
    }
  ]
}
```

## Writing Scenarios

You can write scenario files manually. The minimum required fields are `scenario_id`, `goal`, and `user_attribute`:

```json
{
  "schema_version": "v1",
  "scenarios": [
    {
      "scenario_id": "my-scenario-001",
      "goal": "You want to find out whether your home insurance policy covers water damage from a burst pipe.",
      "user_attribute": {
        "customer_type": "existing customer",
        "location": "Calgary, AB, Canada"
      }
    }
  ]
}
```

<Note>
  **Tips**

  - Write `goal` in second person ("You want to...") so it reads naturally as a simulator instruction.
  - `knowledge` can be omitted if the simulated user has no specific background information.
  - `user_id` defaults to `scenario_id` when omitted.
  - `origin` can be omitted for hand-authored scenarios.
</Note>

## Multi-Knowledge Scenarios

A scenario can include multiple knowledge items by adding additional entries to the `knowledge` list. These items are directly injected into the simulated user's context — there is no retrieval step or external knowledge base involved.

```json
"knowledge": [
  {
    "content": "Home Insurance Basics: TD Insurance provides coverage for your dwelling, contents, and liability...",
    "metadata": { "title": "Home Insurance Basics" }
  },
  {
    "content": "Claims Process: To file a claim, you can call the TD Insurance claims line or use the online claims portal...",
    "metadata": { "title": "How to File a Claim" }
  }
]
```

## How Fields Are Used Downstream

Understanding how each field is used helps you decide what to include.

| Field                  | Used By               | Description                                                          |
| ---------------------- | --------------------- | -------------------------------------------------------------------- |
| `goal`                 | Simulator & Evaluator | Directly used in the simulator prompt; the most important field.     |
| `knowledge[].content`  | Simulator & Evaluator | Provides simulated user's background knowledge for the conversation. |
| `user_attribute`       | Evaluator only        | Creates a persona for evaluating user behavior.                      |
| `knowledge[].metadata` | Not used by Arksim    | For labeling and traceability only.                                  |
| `origin`               | Not used by Arksim    | Tracks scenario provenance; reference only.                          |
